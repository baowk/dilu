## 🎯 Core Principles

This specification is the basic guideline for AI-generated code and must be strictly followed:

1. **Generated Code Must Not Be Modified**: Repository/Model/Query layer code generated by gen command is strictly prohibited from manual modification
2. **Business Logic Concentrated in Service Layer**: All business logic must be implemented in the Service layer
3. **Clear Responsibility Boundaries**: Repository layer handles data persistence, Service layer handles business logic, Api layer encapsulates HTTP interfaces
4. **Safe Iteration Process**: Modify database → Regenerate code → Adapt in Service layer

### Development Advantages

- ⚡ **Rapid Generation**: Automatically generate Model and Query layer code
- 🎨 **Unified Standards**: Follow GORM best practices
- 🔍 **Type Safety**: Strongly typed query operations
- 📝 **Easy Extension**: Easily add business logic on top of generated code
- 🐛 **Error Reduction**: Automated generation reduces human errors

### 📂 Project Directory Overview

The Dilu project contains the following important directories:

#### Core Directories

- **`cmd/`** - Command line tool entry (gen, start, version, etc.)
- **`common/`** - Common utilities (tools, middleware, constants)
- **`internal/`** - Core business logic layer (replaces old modules directory)
  - `bootstrap/` - Application initialization
  - `tools/` - Code generator core implementation
    - `apis/` - API interface layer
    - `service/` - Business logic layer
    - `router/` - Router configuration layer
    - `repository/` - Data access layer
      - `model/` - Model layer
      - `query/` - Query layer
  - `sys/` - System management module (auto-generated + manual extension)
    - `repository/` - Data access layer (Model, Query, generated by gen)
      - `model/` - Model layer (*.gen.go)
      - `query/` - Query layer (*.gen.go)
    - `apis/` - API interface layer (manually created)
    - `service/` - Business logic layer (manually created, core workspace)
    - `router/` - Router configuration layer (manually created)
- **`resources/`** - Configuration files (different environments)
- **`scripts/`** - Scripts (database initialization, migration, etc.)
- **`temp/`** - Temporary files (logs, SQLite database, runtime files)

## 💻 Code Generation Quick Start

### 1. Prepare Database

**Using SQLite (Recommended for Development Testing)**:
```bash
# Initialize SQLite database (with sample data)
go run scripts/init_sqlite.go

# Verify database is created
ls -lh temp/dilu.db
```

**Using MySQL/PostgreSQL (Production Environment)**:
```bash
# 1. Create database
mysql -u root -p -e "CREATE DATABASE dilu_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# 2. Import SQL scripts or manually create tables
mysql -u root -p dilu_db < scripts/dilu.sql

# 3. Update configuration file
vim resources/config.dev.yaml
# Modify dbcfg.driver and dbcfg.dns to actual connection strings
```

### 2. Call gen Command to Generate Code

**Basic Syntax**:
```bash
go run main.go gen \
  -c <config-file> \      # Required: Specify configuration file path
  -t <table-name> \       # Required: Specify database table name
  [-d <database-name>] \  # Optional: Default "sys"
  [-f <true|false>] \     # Optional: Whether to overwrite, default false
  [-p <package-name>]     # Optional: Custom package name
```

**Complete Examples**:

```bash
# Example 1: Generate sys_user table code (SQLite)
go run main.go gen \
  -c resources/config.sqlite.yaml \
  -t sys_user \
  -f false

# Example 2: Generate multiple tables (loop call)
for table in sys_user sys_role sys_dept; do
  go run main.go gen \
    -c resources/config.sqlite.yaml \
    -t $table \
    -f false
done

# Example 3: Force overwrite existing files
go run main.go gen \
  -c resources/config.sqlite.yaml \
  -t sys_user \
  -f true
```

**gen Command Parameters**:

| Parameter | Short | Required | Default | Description | Example |
|-----------|-------|----------|---------|-------------|---------|
| --config | -c | ✅ | resources/config.dev.yaml | Configuration file path | `-c resources/config.sqlite.yaml` |
| --table | -t | ✅ | None | Database table name | `-t sys_user` |
| --db | -d | ❌ | sys | Database name (use when multiple databases) | `-d sys` or omit |
| --force | -f | ❌ | false | Whether to overwrite existing files | `-f true` force overwrite |
| --package | -p | ❌ | Auto-generated | Custom package name | `-p usermodule` |

### 4. Extend on Generated Code

**❌ Wrong Approach**: Directly modify `.gen.go` files
```go
// ❌ Don't do this!
// internal/sys/repository/model/sys_user.gen.go
type SysUser struct {
    CustomField string  // ⚠️ Will be lost on regeneration
}
```

**✅ Correct Approach**: Create extension file
```go
// ✅ Recommended: Create separate extension file
// internal/sys/repository/model/sys_user_extend.go
package model

// SysUserWithRoles extends user with roles
type SysUserWithRoles struct {
    SysUser
    Roles []SysRole `json:"roles" gorm:"many2many:sys_user_role;"`
}

// GetUserWithRoles gets user with roles
func GetUserWithRoles(userID uint) (*SysUserWithRoles, error) {
    // Implement complex query
    var user SysUserWithRoles
    err := db.Preload("Roles").First(&user, userID).Error
    return &user, err
}
```

### Unified Response Format

```go
// Prompt example:
// "Define unified API response format including success response, error response, pagination response"

package base

type Response struct {
    Code    int         `json:"code"`
    Message string      `json:"message"`
    Data    interface{} `json:"data"`
}

type PageResponse struct {
    List     interface{} `json:"list"`
    Total    int64       `json:"total"`
    Page     int         `json:"page"`
    PageSize int         `json:"pageSize"`
}

func Success(data interface{}) Response {
    return Response{
        Code:    200,
        Message: "success",
        Data:    data,
    }
}

func Error(code int, message string) Response {
    return Response{
        Code:    code,
        Message: message,
        Data:    nil,
    }
}
```
