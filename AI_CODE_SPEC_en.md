## 🎯 Core Principles (Must Follow)

1. **Prefer gen for code generation** - Keep code structure clean and organized
2. **Use TDD development mode** - Test-first approach
3. **Generated code must not be modified** - Repository/Model/Query layer code generated by gen command is strictly prohibited from manual modification
4. **Business logic in Service layer** - All business logic must be implemented in the Service layer
5. **Clear responsibility boundaries** - Repository handles data, Service handles business, Api handles interfaces
6. **Safe iteration** - Modify database → Regenerate → Adapt in Service layer
7. **Use structured logging** - Use log/slog
8. **Don't reinvent the wheel** - Standard library first, third-party libraries used cautiously
9. **Swagger documentation** - APIs use Swagger for documentation with clear annotations, generated via go generate
10. **Follow Golang code conventions** - Version Go 1.26+

## 📋 Basic Specifications

### Project Structure
```
dilu/
├── cmd/                    # Command line tools
├── common/                 # Common utilities
├── internal/               # Core business
│   ├── bootstrap/         # Initialization
│   ├── tools/             # Code generator
│   │   ├── apis/          # API interface
│   │   ├── service/       # Business logic
│   │   ├── router/        # Router config
│   │   └── repository/    # Data access ⭐
│   │       ├── model/     # Model layer
│   │       └── query/     # Query layer
│   └── sys/               # Generated by gen, default: sys
├── resources/              # Config files
├── docs/                   # User docs
├── dev-docs/               # Dev docs
│   └── releases/          # Release docs
└── tests/                  # Test code
```

### Layer Responsibilities
| Layer | Responsibility | File Example |
|-------|----------------|--------------|
| **Repository** | Data persistence | `*.gen.go` (DO NOT MODIFY, regenerate when fields change, will overwrite by default) |
| **Service** | Business logic | `sys_user_service.go` |
| **Api** | HTTP interface wrapper | `sys_user_api.go` |
| **Router** | Route configuration | `sys_user_router.go` |

### Naming Conventions
- **Model**: PascalCase, e.g., `SysUser`
- **Method**: camelCase, e.g., `GetUserByID`
- **Variable**: camelCase, e.g., `userName`
- **Constant**: UPPER_CASE, e.g., `TableNameSysUser`
- **Package**: lowercase, e.g., `model`, `query`, `service`

### Error Handling
```go
// ✅ Correct: Return error immediately
if err != nil {
    return nil, err
}

// ✅ Correct: Wrap error with context
if err != nil {
    return fmt.Errorf("get user failed: %w", err)
}

// ❌ Wrong: Ignore error
getUser(id) // err not handled
```

### Logging Standards
```go
// ✅ Correct: Structured logging (using slog)
slog.Debug("get user success", "user", user)   // Debug level
slog.Info("get user success", "user", user)    // Info level
slog.Warn("get user timeout", "user", user)    // Warn level
slog.Error("get user failed", "error", err)    // Error level

// ❌ Wrong: Meaningless log
log.Println("error occurred")
```

## 💻 Code Generation Flow (Multi-Database Architecture)

Dilu supports multi-database architecture. You can specify different databases using the `-d` parameter.

### Step 1: Prepare Database
```bash
# SQLite (Recommended for development)
go run scripts/init_sqlite.go

# MySQL (Production)
mysql -u root -p -e "CREATE DATABASE dilu_db"
```

### Step 2: Run Generator

#### Example 1: Generate sys_user table from system database (default)
```bash
go run main.go gen \
  -c resources/config.sqlite.yaml \
  -t sys_user
```

#### Example 2: Generate message table from notice database (specified)
```bash
go run main.go gen \
  -c resources/config.sqlite.yaml \
  -d notice \
  -t message
```

**Parameters**:
- `-c`: Configuration file path
- `-d`: Database name (optional, default: `sys`)
- `-t`: Table name
- `-f`: Overwrite existing files (optional)
- `-p`: Package name (optional, defaults to same as `-d` parameter)

### Step 3: Verify Generation
```bash
# Check generated files
ls internal/sys/repository/model/sys_user.gen.go
ls internal/sys/repository/query/sys_user.gen.go
```

### Step 4: Extend Implementation
```go
// ✅ Create extension file (DO NOT modify .gen.go)
// internal/sys/repository/model/sys_user_extend.go
package model

type SysUserWithRoles struct {
    SysUser
    Roles []SysRole `json:"roles"`
}
```

## 🌐 API Development Standards

### RESTful Route Design
```go
// QueryPage - POST /api/v1/sys/user/page     - Paginated query
// Get      - GET  /api/v1/sys/user/{id}      - Get single record
// Create   - POST /api/v1/sys/user/create    - Create record
// Update   - POST /api/v1/sys/user/update    - Update record
// Del      - POST /api/v1/sys/user/del       - Delete record
```

### Unified Response Format
```go
// Success response (with data)
e.Ok(c, data)
// Returns: {"code": 200, "message": "success", "data": {...}}

// Success response (no data)
e.Ok(c)
// Returns: {"code": 200, "message": "success"}

// Paginated response
e.Page(c, list, total, page, pageSize)
// Returns: {"code": 200, "data": {"list": [...], "total": 100}}

// Error response
e.Error(c, err)
// Returns: {"code": 500, "message": "error message"}
```

## 🗄️ Database Design

### Table Design Standards
```sql
-- User table
CREATE TABLE `sys_user` (
  `id` int NOT NULL AUTO_INCREMENT COMMENT 'User ID',
  `username` varchar(100) NOT NULL COMMENT 'Username',
  `password` varchar(255) NOT NULL COMMENT 'Password',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_username` (`username`)
) ENGINE=InnoDB COMMENT='User table';
```

### GORM Model Standards
Auto-generated by GORM, DO NOT modify manually

```go
// Example model structure
type SysUser struct {
    ID        int       `gorm:"primarykey" json:"id"`
    Username  string    `gorm:"uniqueIndex" json:"username"`
    Password  string    `json:"password"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}
```

## ⭐ Code Elegance Principles

#### ✅ DO
- **Simple & Direct**: Clear logic, easy to understand at a glance
- **Single Responsibility**: Each function does one thing only
- **Accurate Naming**: Self-explanatory, no comments needed
- **Error Handling**: Return immediately, avoid nesting
- **Code Reuse**: Extract common logic, avoid duplication

#### ❌ DON'T
- **Over-engineering**: Don't use complex patterns for showing off
- **Clever Programming**: Avoid obscure syntax sugar and tricks
- **Premature Optimization**: Make it work first, then optimize
- **Abuse of Abstraction**: Don't abstract for abstraction's sake
- **Magic Numbers**: Use meaningful constants

## ✅ TDD Programming Practices

### Test-First Flow
```
1. Write failing test → 2. Write minimal implementation → 3. Refactor
```

### Test Coverage Requirements
- **Service Layer**: Core business logic coverage ≥ 80%
- **Api Layer**: Main interface coverage ≥ 70%
